```javascript
// Pseudocódigo del proceso
async function migrarProductos(archivoExcel) {
  const productos = leerExcel(archivoExcel);
  
  for (const producto of productos) {
    // Validar datos
    if (!producto.nombre || !producto.precio) {
      throw new Error(`Producto inválido: ${producto}`);
    }
    
    // Normalizar
    producto.nombre = normalizarTexto(producto.nombre);
    producto.precio = parseFloat(producto.precio);
    producto.stock = parseInt(producto.stock) || 0;
    
    // Insertar en Supabase
    await supabase.from('productos').insert(producto);
  }
  
  console.log(`✅ ${productos.length} productos migrados`);
}
```

**Validaciones post-carga:**

#### 2.3 Ajuste de Inventario Actual
**Duración:** 1 día  
**Prioridad:** 🔴 CRÍTICA

**Este es el stock REAL actual en bodega física.**

**Proceso:**
1. Hacer conteo físico completo de inventario
2. Comparar con stock en tabla `productos`
3. Ajustar diferencias mediante movimientos de inventario

**Script de ajuste:**
```javascript
async function ajustarInventario(productoId, stockFisico, motivo) {
  // Obtener stock actual en sistema
  const { data: producto } = await supabase
    .from('productos')
    .select('stock')
    .eq('id', productoId)
    .single();
  
  const diferencia = stockFisico - producto.stock;
  
  if (diferencia !== 0) {
    // Registrar movimiento de ajuste
    await supabase.from('movimientos_inventario').insert({
      producto_id: productoId,
      tipo_movimiento: 'ajuste',
      cantidad: Math.abs(diferencia),
      stock_anterior: producto.stock,
      stock_nuevo: stockFisico,
      descripcion: `Ajuste inicial: ${motivo}`,
      usuario: 'Sistema - Migración',
      fecha_movimiento: new Date().toISOString()
    });
    
    // Actualizar stock en productos
    await supabase
      .from('productos')
      .update({ stock: stockFisico })
      .eq('id', productoId);
    
    console.log(`✅ Stock ajustado: ${producto.stock} → ${stockFisico}`);
  }
}
```

**Validaciones:**
- [ ] Todos los productos tienen stock ≥ 0
- [ ] Todos los ajustes están registrados en movimientos_inventario
- [ ] Stock físico = stock en sistema

```sql
  p->>'nombre' as nombre_producto

-- 3. Verificar abonos con facturas inexistentes

FROM abonos a
LEFT JOIN facturas f ON a.factura_id = f.id
WHERE f.id IS NULL;
-- Resultado esperado: 0 registros

-- 4. Verificar sumas de abonos vs total factura
SELECT 
  f.id,
  f.total,
  COALESCE(SUM(a.monto), 0) as total_abonado,
  f.total - COALESCE(SUM(a.monto), 0) as saldo
FROM facturas f
LEFT JOIN abonos a ON f.id = a.factura_id
GROUP BY f.id
HAVING SUM(a.monto) > f.total;
-- Resultado esperado: 0 registros (no hay sobreabonos)
```

## 🛡️ PLAN DE CONTINGENCIA Y RIESGOS

### Matriz de Riesgos

| Riesgo | Probabilidad | Impacto | Mitigación | Plan B |
|--------|--------------|---------|------------|--------|
| **Datos origen incompletos** | 🟡 Media | 🔴 Alto | Validación temprana en Fase 0 | Completar manualmente con cliente |
| **Corrupción durante carga** | 🟢 Baja | 🔴 Alto | Backups antes de cada carga | Restaurar desde backup + reintentar |
| **Usuarios no capacitados** | 🟡 Media | 🟡 Medio | Capacitación + manuales | Soporte extendido primera semana |
| **Pérdida de facturas físicas** | 🟢 Baja | 🔴 Alto | Digitalizar/fotografiar al recibir | Aceptar missing data (documentado) |
| **Downtime prolongado** | 🟢 Baja | 🟡 Medio | Cutover en horario no laboral | Rollback + reprogramar |
| **Resistencia al cambio** | 🟡 Media | 🟡 Medio | Gestión del cambio + involucrar usuarios | Acompañamiento personalizado |

---

### Estrategia de Backups

**Frecuencia durante migración:**
```
Fase 0: Diario (staging)
Fase 1: Antes/después de cada tabla migrada
Fase 2: Cada 4 horas durante carga transaccional
Fase 3: Diario
Fase 4: Cada hora durante cutover
Fase 5: Diario (primera semana) → Semanal (normal)
```

**Tipos de backup:**
1. **Backup completo:** Base de datos completa
2. **Backup incremental:** Solo cambios desde último backup
3. **Backup de aplicación:** Código y configuración

**Retención:**
- Backups durante migración: 30 días
- Backups post go-live: 90 días
- Backup pre-cutover: 1 año (referencia)
