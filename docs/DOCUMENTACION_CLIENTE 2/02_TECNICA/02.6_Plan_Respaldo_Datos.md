## 🎯 OBJETIVO Y ALCANCE

### Objetivo Principal
Garantizar la **disponibilidad, integridad y recuperabilidad** de todos los datos críticos del Sistema EBS, minimizando la pérdida de información ante cualquier incidente y asegurando la continuidad del negocio.

### Objetivos Específicos
✅ **RPO (Recovery Point Objective):** Pérdida máxima de datos aceptable = **1 hora**  
✅ **RTO (Recovery Time Objective):** Tiempo máximo de recuperación = **4 horas**  
✅ **Disponibilidad del sistema:** 99.5% anual (downtime máximo ~43 horas/año)  
✅ **Integridad de datos:** 100% verificada mediante checksums  
✅ **Retención histórica:** Mínimo 90 días, críticos 1 año

### Alcance del Plan

**Incluye:**
- ✅ Base de datos completa (Supabase)
- ✅ Archivos de configuración
- ✅ Imágenes de productos (Cloudinary)
- ✅ Documentación del sistema
- ✅ Código fuente de la aplicación

**No incluye (fuera de alcance):**
- ❌ Sistemas operativos del servidor (responsabilidad de Supabase)
- ❌ Software de terceros (Cloudinary hace sus propios backups)
- ❌ Estaciones de trabajo locales

## 📜 POLÍTICA DE RESPALDOS

### Principios Fundamentales

#### 1. **Regla 3-2-1**
```
3 copias de los datos:
  ├─ 1 copia primaria (producción)
  ├─ 1 copia local/nearline (Supabase backups)
  └─ 1 copia remota/offsite (AWS S3 / Google Cloud Storage)

2 tipos de medios diferentes:
  ├─ Base de datos Supabase
  └─ Almacenamiento objeto (S3/GCS)

1 copia fuera del sitio (off-site):
  └─ Región geográfica diferente
```

#### 2. **Criticidad de Datos**

| Nivel | Tipo de Dato | RPO | RTO | Retención |
|-------|--------------|-----|-----|-----------|
| 🔴 **CRÍTICO** | Facturas, transacciones | 1 hora | 2 horas | 1 año |
| 🟡 **IMPORTANTE** | Productos, clientes | 4 horas | 4 horas | 90 días |
| 🟢 **NORMAL** | Configuración, logs | 24 horas | 8 horas | 30 días |

#### 3. **Responsabilidad Compartida**

| Componente | Responsable | Backup | Verificación |
|------------|-------------|--------|--------------|
| Base de datos Supabase | Supabase (automático) | Continuo | Diario |
| Backup personalizado | Edwin Marín / EBS | Diario | Semanal |
| Imágenes Cloudinary | Cloudinary (automático) | Continuo | Mensual |
| Código fuente | GitHub | Cada commit | Automático |
| Backup off-site | EBS | Semanal | Mensual |

## 📦 TIPOS DE RESPALDO

### 1. **Backup Completo (Full Backup)**

**Descripción:** Copia completa de toda la base de datos y archivos.

**Ventajas:**
- ✅ Recuperación más rápida
- ✅ Independiente de otros backups
- ✅ Integridad garantizada

**Desventajas:**
- ❌ Consume más espacio
- ❌ Toma más tiempo

**Cuándo se ejecuta:**
- Diario: 2:00 AM (automatizado)
- Semanal: Domingos 3:00 AM (off-site)
- Manual: Antes de cambios críticos

**Contenido:**
```sql
-- Tablas incluidas en backup completo
✅ productos (todos los campos)
✅ clientes (todos los campos)
✅ facturas (incluye JSONB de productos)
✅ abonos (todos los registros)
✅ movimientos_inventario (historial completo)
✅ auditoria_productos (logs de cambios)
✅ pedidos (si existe)
✅ usuarios (credenciales hasheadas)
```

---

### 2. **Backup Incremental**

**Descripción:** Solo registros modificados desde el último backup.

**Ventajas:**
- ✅ Rápido de ejecutar
- ✅ Ahorra espacio
- ✅ Frecuencia mayor posible

**Desventajas:**
- ❌ Recuperación más lenta
- ❌ Requiere backup completo previo

**Cuándo se ejecuta:**
- Cada 6 horas: 8:00 AM, 2:00 PM, 8:00 PM
- Automático (cron job)

**Query de ejemplo:**
```sql
-- Backup incremental: solo cambios desde última backup
SELECT * FROM facturas 
WHERE created_at > (
  SELECT MAX(fecha_backup) 
  FROM control_backups 
  WHERE tipo = 'incremental'
);

SELECT * FROM productos 
WHERE updated_at > [ULTIMO_BACKUP];
```

---

### 3. **Backup Diferencial**

**Descripción:** Cambios desde el último backup COMPLETO.

**Cuándo se usa:**
- Antes de actualizaciones mayores del sistema
- Antes de migraciones de datos
- Antes de cambios masivos en catálogo

**Ventajas:**
- ✅ Balance entre completo e incremental
- ✅ Recuperación más rápida que incremental

---

### 4. **Snapshot de Base de Datos**

**Descripción:** Imagen instantánea del estado de la BD (Supabase nativo).

**Características:**
- Automático por Supabase
- Cada hora
- Retención: 7 días
- Recuperación en < 30 minutos

**Uso:**
- Recuperación rápida de errores recientes
- Rollback de cambios accidentales
- No requiere restauración completa

---

### 5. **Backup Lógico vs Físico**

| Tipo | Descripción | Herramienta | Uso |
|------|-------------|-------------|-----|
| **Lógico** | Exporta datos como SQL/JSON | `pg_dump`, scripts custom | Portabilidad, migración |
| **Físico** | Copia archivos de BD raw | Supabase snapshots | Recuperación rápida |

**Estrategia EBS:** Usamos **AMBOS**
- Lógico: Backup diario personalizado (portabilidad)
- Físico: Supabase snapshots (velocidad)

## 🔧 PROCEDIMIENTOS DE RESPALDO

### Procedimiento 1: Backup Completo Automático Diario

**Frecuencia:** Diario a las 2:00 AM  
**Duración estimada:** 5-10 minutos  
**Responsable:** Sistema (cron job)

**Script automatizado:**

```bash
#!/bin/bash
# Script: backup_diario_ebs.sh
# Ubicación: /opt/ebs/scripts/backup_diario_ebs.sh

# Variables de configuración
FECHA=$(date +%Y-%m-%d_%H-%M-%S)
BACKUP_DIR="/var/backups/ebs"
LOG_FILE="/var/log/ebs_backup.log"
SUPABASE_URL="https://jqkfykverasqwlfsjsnj.supabase.co"
SUPABASE_KEY="[ANON_KEY]"
S3_BUCKET="s3://ebs-backups-prod"

# Función de log
log() {
    echo "[$(date +'%Y-%m-%d %H:%M:%S')] $1" | tee -a $LOG_FILE
}

log "=== INICIO BACKUP COMPLETO DIARIO ==="

# 1. Crear directorio de backup del día
mkdir -p "${BACKUP_DIR}/${FECHA}"
cd "${BACKUP_DIR}/${FECHA}"

# 2. Backup de cada tabla en formato JSON
log "Exportando tablas..."

# Productos
curl -X GET "${SUPABASE_URL}/rest/v1/productos?select=*" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}" \
  > productos.json
log "✓ Productos exportados: $(wc -l < productos.json) registros"

# Clientes
curl -X GET "${SUPABASE_URL}/rest/v1/clientes?select=*" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}" \
  > clientes.json
log "✓ Clientes exportados"

# Facturas
curl -X GET "${SUPABASE_URL}/rest/v1/facturas?select=*" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}" \
  > facturas.json
log "✓ Facturas exportadas"

# Abonos
curl -X GET "${SUPABASE_URL}/rest/v1/abonos?select=*" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}" \
  > abonos.json
log "✓ Abonos exportados"

# Movimientos inventario
curl -X GET "${SUPABASE_URL}/rest/v1/movimientos_inventario?select=*" \
  -H "apikey: ${SUPABASE_KEY}" \
  -H "Authorization: Bearer ${SUPABASE_KEY}" \
  > movimientos_inventario.json
log "✓ Movimientos de inventario exportados"

# 3. Crear archivo de metadatos
cat > metadata.json <<EOF
{
  "fecha_backup": "${FECHA}",
  "tipo": "completo_diario",
  "version_sistema": "1.0",
  "tablas": ["productos", "clientes", "facturas", "abonos", "movimientos_inventario"],
  "servidor": "$(hostname)",
  "usuario": "$(whoami)"
}
EOF

# 4. Comprimir backup
log "Comprimiendo backup..."
tar -czf "backup_${FECHA}.tar.gz" *.json metadata.json
BACKUP_SIZE=$(du -h "backup_${FECHA}.tar.gz" | cut -f1)
log "✓ Backup comprimido: ${BACKUP_SIZE}"

# 5. Verificar integridad (checksum)
sha256sum "backup_${FECHA}.tar.gz" > "backup_${FECHA}.sha256"
log "✓ Checksum generado"

# 6. Limpiar archivos temporales
rm -f *.json

# 7. Eliminar backups antiguos (>30 días)
log "Limpiando backups antiguos..."
find ${BACKUP_DIR} -name "backup_*.tar.gz" -mtime +30 -delete
log "✓ Backups antiguos eliminados"

# 8. Registrar en base de datos
log "Registrando backup en sistema..."
# [Aquí iría el registro en tabla control_backups]

# 9. Notificación de éxito
log "=== BACKUP COMPLETADO EXITOSAMENTE ==="
log "Tamaño: ${BACKUP_SIZE}"
log "Ubicación: ${BACKUP_DIR}/${FECHA}/backup_${FECHA}.tar.gz"

# Opcional: Enviar notificación por email/webhook
# curl -X POST https://hooks.slack.com/... -d "Backup diario completado"

exit 0
```

**Configuración del cron:**
```bash
# Editar crontab
crontab -e

# Agregar línea para ejecutar a las 2 AM diario
0 2 * * * /opt/ebs/scripts/backup_diario_ebs.sh >> /var/log/ebs_backup.log 2>&1
```

```bash
log "=== BACKUP INCREMENTAL COMPLETADO ==="
```

### Procedimiento 3: Backup Off-Site Semanal

**Frecuencia:** Domingos 3:00 AM  
**Duración estimada:** 15-20 minutos  
**Destino:** AWS S3 (región us-west-2)

**Script:**
```bash
#!/bin/bash
# Script: backup_offsite_semanal.sh

FECHA=$(date +%Y-%m-%d)
BACKUP_LOCAL="/var/backups/ebs"
S3_BUCKET="s3://ebs-backups-prod"
S3_PATH="${S3_BUCKET}/weekly/${FECHA}"

log "=== INICIO BACKUP OFF-SITE SEMANAL ==="

# 1. Ejecutar backup completo primero
/opt/ebs/scripts/backup_diario_ebs.sh

# 2. Sincronizar a S3
log "Sincronizando con AWS S3..."
aws s3 sync ${BACKUP_LOCAL} ${S3_PATH} \
  --storage-class STANDARD_IA \
  --exclude "incremental/*" \
  --exclude "*.log"

# 3. Verificar sincronización
S3_COUNT=$(aws s3 ls ${S3_PATH} --recursive | wc -l)
log "✓ ${S3_COUNT} archivos sincronizados a S3"

# 4. Prueba de recuperación (descargar y verificar checksum)
log "Verificando integridad en S3..."
ULTIMO_BACKUP=$(ls -t ${BACKUP_LOCAL}/*/backup_*.tar.gz | head -1)
CHECKSUM_LOCAL=$(sha256sum ${ULTIMO_BACKUP} | cut -d' ' -f1)

aws s3 cp ${S3_PATH}/$(basename ${ULTIMO_BACKUP}).sha256 /tmp/
CHECKSUM_S3=$(cat /tmp/$(basename ${ULTIMO_BACKUP}).sha256 | cut -d' ' -f1)

if [ "${CHECKSUM_LOCAL}" == "${CHECKSUM_S3}" ]; then
    log "✓ Integridad verificada correctamente"
else
    log "✗ ERROR: Checksums no coinciden"
    # Enviar alerta crítica
    exit 1
fi

# 5. Configurar lifecycle policy para mover a Glacier después de 90 días
aws s3api put-bucket-lifecycle-configuration \
  --bucket ebs-backups-prod \
  --lifecycle-configuration file:///opt/ebs/config/s3_lifecycle.json

log "=== BACKUP OFF-SITE COMPLETADO ==="
```

## 🔄 PROCEDIMIENTOS DE RESTAURACIÓN

### Escenarios de Recuperación

#### Escenario 1: Recuperación de Factura Eliminada Accidentalmente

**RPO:** < 1 hora  
**RTO:** < 15 minutos  
**Complejidad:** Baja

**Pasos:**

1. **Identificar factura a recuperar:**
   ```sql
   -- Buscar en tabla de auditoría
   SELECT * FROM auditoria_facturas 
   WHERE factura_id = [ID] 
   ORDER BY fecha_eliminacion DESC LIMIT 1;
   ```

2. **Usar Point-in-Time Recovery de Supabase:**
   - Dashboard Supabase → Database → Backups
   - Seleccionar timestamp antes de eliminación
   - Clic en "Restore specific tables"
   - Seleccionar tabla `facturas`
   - Especificar ID de factura

3. **Verificar recuperación:**
   ```sql
   SELECT * FROM facturas WHERE id = [ID];
   ```

4. **Notificar al usuario:**
   ```
   "Factura #[ID] recuperada exitosamente.
   Datos restaurados al: [fecha/hora]"
   ```

**Tiempo estimado:** 10-15 minutos

---

#### Escenario 2: Corrupción de Tabla de Productos

**RPO:** 4 horas (último incremental)  
**RTO:** 1 hora  
**Complejidad:** Media

**Pasos:**

1. **Confirmar corrupción:**
   ```sql
   SELECT COUNT(*) FROM productos; -- Error o resultado incorrecto
   ```

2. **Poner tabla en modo solo lectura:**
   ```sql
   -- Crear política temporal
   CREATE POLICY "readonly" ON productos FOR ALL USING (false);
   ```

3. **Descargar último backup completo:**
   ```bash
   cd /tmp
   aws s3 cp s3://ebs-backups-prod/latest/backup_productos.json .
   ```

4. **Truncar tabla corrupta:**
   ```sql
   TRUNCATE TABLE productos CASCADE; -- Cuidado con dependencias
   ```

5. **Importar desde backup:**
   ```bash
   # Usando script de importación
   node /opt/ebs/scripts/import_productos.js backup_productos.json
   ```

6. **Verificar integridad:**
   ```sql
   SELECT COUNT(*) FROM productos; -- Debe coincidir con backup
   SELECT * FROM productos WHERE precio IS NULL; -- No debe haber
   ```

7. **Restaurar políticas normales:**
   ```sql
   DROP POLICY "readonly" ON productos;
   ```

8. **Aplicar cambios incrementales (si los hay):**
   ```bash
   # Importar cambios desde último incremental
   node /opt/ebs/scripts/import_incremental.js
   ```

**Tiempo estimado:** 45-60 minutos

---

#### Escenario 3: Pérdida Total de Base de Datos

**RPO:** 24 horas (peor caso)  
**RTO:** 4 horas  
**Complejidad:** Alta

**Pasos:**

1. **⚠️ ACTIVAR PLAN DE RECUPERACIÓN DE DESASTRES**

2. **Evaluar daño:**
   - ¿Supabase disponible? → Restaurar desde snapshot
   - ¿Supabase caído? → Restaurar desde backup off-site

3. **Opción A: Restaurar desde Supabase Snapshot**
   ```
   Dashboard Supabase → Database → Backups
   → Select latest snapshot
   → Click "Restore"
   → Confirm (takes 15-30 min)
   ```

4. **Opción B: Restaurar desde AWS S3 (desastre total)**

   a. **Crear nueva instancia Supabase:**
      - Dashboard → New Project
      - Region: us-east-1
      - Plan: Pro (mínimo)

   b. **Descargar último backup semanal:**
      ```bash
      aws s3 sync s3://ebs-backups-prod/weekly/latest /tmp/restore
      ```

   c. **Descomprimir:**
      ```bash
      cd /tmp/restore
      tar -xzf backup_*.tar.gz
      ```

   d. **Verificar integridad:**
      ```bash
      sha256sum -c backup_*.sha256
      ```

   e. **Crear estructura de tablas:**
      ```bash
      psql -h [NEW_SUPABASE_HOST] -U postgres -f /opt/ebs/schema/create_tables.sql
      ```

   f. **Importar datos:**
      ```bash
      node /opt/ebs/scripts/restore_full.js /tmp/restore
      ```

   g. **Verificar conteos:**
      ```sql
      SELECT 'productos' as tabla, COUNT(*) FROM productos
      UNION ALL
      SELECT 'clientes', COUNT(*) FROM clientes
      UNION ALL
      SELECT 'facturas', COUNT(*) FROM facturas;
      ```

   h. **Actualizar variables de entorno en aplicación:**
      ```bash
      export VITE_SUPABASE_URL="https://[NEW_PROJECT].supabase.co"
      export VITE_SUPABASE_ANON_KEY="[NEW_ANON_KEY]"
      ```

   i. **Aplicar incrementales (si existen):**
      ```bash
      # Restaurar cambios desde último semanal
      for file in /tmp/restore/incremental/*.json; do
        node /opt/ebs/scripts/import_incremental.js $file
      done
      ```

   j. **Pruebas de humo:**
      - [ ] Login funciona
      - [ ] Crear factura de prueba

### Script de Monitoreo Automático

```bash
#!/bin/bash
# Script: monitor_backups.sh
# Ejecuta cada hora para verificar estado de backups

ULTIMO_BACKUP=$(ls -t /var/backups/ebs/*/backup_*.tar.gz | head -1)
EDAD_BACKUP=$(( ($(date +%s) - $(stat -f %m "$ULTIMO_BACKUP")) / 3600 ))

# Alerta si backup más reciente tiene >25 horas
if [ $EDAD_BACKUP -gt 25 ]; then
    echo "🚨 ALERTA: Último backup tiene $EDAD_BACKUP horas de antigüedad"
    # Enviar webhook a Slack/Email
    curl -X POST https://hooks.slack.com/services/XXX/YYY/ZZZ \
      -d "{\"text\": \"⚠️ Backup EBS retrasado: ${EDAD_BACKUP} horas\"}"
fi

# Verificar tamaño del backup (debe ser >1MB)
TAMANIO=$(stat -f %z "$ULTIMO_BACKUP")
if [ $TAMANIO -lt 1048576 ]; then
    echo "🚨 ALERTA: Backup sospechosamente pequeño: $(($TAMANIO / 1024))KB"
fi

# Verificar espacio en disco
ESPACIO_LIBRE=$(df -h /var/backups | tail -1 | awk '{print $5}' | sed 's/%//')
if [ $ESPACIO_LIBRE -gt 90 ]; then
    echo "⚠️ ADVERTENCIA: Disco casi lleno: ${ESPACIO_LIBRE}%"
fi

# Verificar conectividad con S3
aws s3 ls s3://ebs-backups-prod/ > /dev/null 2>&1
if [ $? -ne 0 ]; then
    echo "🚨 ALERTA: No se puede acceder a S3"
fi

echo "✓ Monitoreo completado: $(date)"
```

**Configurar en cron cada hora:**
```bash
0 * * * * /opt/ebs/scripts/monitor_backups.sh >> /var/log/monitor_backups.log 2>&1
```
